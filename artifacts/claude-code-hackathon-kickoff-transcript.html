Boris Tcherny's Lessons from Building Claude Code
Lesson 01

Build for the model 6 months from now

Claude Code wasn't an instant hit at launch in February. It went mainstream in May when Sonnet 4 and Opus 4 arrived. Boris started building in September 2024 on Sonnet 3.5, deliberately building for the future model so the product would click when the model caught up.

Don't build for Opus 4.6 -- build for the model six months after Opus 4.6. Feel out the boundaries of what the model can do; those things will improve.
Lesson 02

Think in terms of latent demand

Go to where the users are. Don't try to make people do a new thing -- make the existing thing they're doing a little easier. The terminal survived as a UI because it's where engineers already live.

The same applies to models: look at what the model is trying to do, and redesign your tools to align with that. In model training, this is called being on distribution.

If you try to make the model do something it doesn't want to do, you're not gonna have a good time.
Lesson 03

Automate everything

At Anthropic, Claude Code handles: code review (running in CI on GitHub, open source), GitHub issue labeling and triage, de-duping issues, feedback triage, security review, and production launches.

The best way: use the Claude Agent SDK -- the same agent under Claude Code's hood, available as TypeScript or Python. Or use claude -p or the Anthropic API directly.

Anything tedious, repetitive, or that you don't want to do yourself -- just automate it.
